# 토스 자동매매 실행 레이어 설계 (toss_execution.py)

작성일: 2026-06-23
상태: 설계 승인됨 → 구현 계획 단계

## 목표

scanner_v4.py가 만든 `signals.json`을 읽어 **토스증권 실계좌에 주문을 실행**하는
레이어. **기본은 드라이런(모의실행)** — 실제 주문은 명시적 `--live` 플래그가 있을 때만.

토스 Open API는 샌드박스/페이퍼 모드가 없으므로(실거래 서버 단일), 안전 검증은
"우리 코드 안의 드라이런"으로 대체한다.

## 배경 / 제약

- 토스 Open API = 실거래뿐 (sandbox 0). 따라서 실주문은 최후 단계.
- 기존 `execution_layer.py`(Alpaca 페이퍼)는 건드리지 않는다 — 토스 경로를 별도 파일로
  분리해 검증 끝날 때까지 Alpaca 경로를 보존.
- 백테스트 결론: 절대수익은 생존편향으로 부풀려짐 → 실거래 전환은 신중하게, 소액부터.
- 보유 7종목은 모두 `strategy: Core` → 자동 매도 대상 아님. 실제 행동은 주로 신규 진입.

## 아키텍처

```
scanner_v4.py → signals.json → toss_execution.py
                                  ├ TossClient: 실보유/예수금/현재가/호가 조회 (구현됨)
                                  ├ 안전장치 검사
                                  ├ 주문 수량 계산 (execution_layer 로직 재사용)
                                  └ 모드 분기:
                                     · 드라이런(기본): 의도 주문을 로그 + 디스코드, 주문 X
                                     · --live: TossClient.create_order() 실제 주문 (2단계)
```

### 컴포넌트

1. **`toss_execution.py`** (신규) — 오케스트레이터
   - 입력: `signals.json` (execution_layer.py와 동일 인터페이스)
   - 토스 실보유(`get_holdings`)·예수금(`get_buying_power`)과 대조 — 실제 보유분에만 매도 적용
   - 각 신호에 안전장치 검사 후, 모드에 따라 로그/알림 또는 실주문
   - 출력: 디스코드 리포트 + 체결/의도 이력 로그 (`toss_execution_history.json`)

2. **`TossClient.create_order()`** (2단계에서 신규) — `POST /api/v1/orders` 래퍼.
   1단계 드라이런에는 불필요(호출 경로 자체를 안 켬). 2단계에서 추가하고 `--live`로만 호출.

### 안전장치 (execution_layer.py에서 이식 + 토스 특화)

- **킬스위치**: `kill_switch.flag` 존재 시 즉시 전면 중단
- **일일 손실 한도**: 토스 계좌 평가액이 당일 시작 대비 −3% 초과 시 신규 진입 차단
- **포지션 캡**: 전략 A/B 각 최대 10개
- **가격 sanity**: 스캐너 가격 vs 토스 실시간가 괴리 과대 시 해당 주문 스킵 (order_preflight 로직)
- **드라이런 기본**: `--live` 없으면 절대 실주문 안 함
- **장 운영 확인**: 정규장 아니면 주문 보류 (order_preflight의 market-calendar)

## 데이터 흐름

1. `signals.json` 로드 (날짜 검증 — 오래된 신호 경고)
2. 킬스위치 확인 → 있으면 중단
3. 토스 실보유·예수금 조회, 당일 시작 평가액 기록
4. **청산(exits)** 처리: 실제 보유 + Core 아님인 종목만 매도 신호 적용
5. 일일 손실 한도 검사 → 초과 시 이하 신규진입 스킵
6. **진입(entries A/B/C/D)** 처리: 포지션 캡·예수금·가격 sanity·장운영 통과분만
7. 각 주문: 드라이런이면 의도 기록, --live면 create_order
8. 디스코드 리포트 + 이력 저장

## 에러 처리

- 토스 조회 실패(네트워크/401/IP) → 해당 주문 스킵하고 계속, 경고 로그
- create_order 실패 → 이력에 실패 기록 + 디스코드 경고, 다음 주문 계속
- signals.json 없음/날짜 불일치 → 경고 후 중단(오래된 신호로 매매 방지)

## 단계 (Phasing)

1. **드라이런** — `toss_execution.py`, 실주문 경로 비활성. 신호→안전검사→의도 알림.
   몇 주 수동 검증.
2. **실주문** — `TossClient.create_order()` + `--live` 소액. 킬스위치 실테스트.
3. **스케줄링** — launchd 평일 장마감 후 자동.

별도(선택): 검증된 개선 `B_ATR_MULT=4`를 config에 반영.

## 테스트

- 드라이런 단계: signals.json 샘플로 의도 주문이 안전장치를 정확히 통과/차단하는지 확인
  (예: 예수금 부족 시 진입 스킵, Core 종목 매도 안 함, 킬스위치 동작, 장 마감 시 보류)
- 실주문 단계 전: create_order를 최소 1주 소액으로 수동 1회 검증

## 범위 밖 (YAGNI)

- 1단계에서 실주문 코드(create_order) 미구현
- 국내(KRX) 종목 자동매매 (스캐너가 US 유니버스라 해당 없음)
- 복잡한 분할 체결/지정가 최적화 — 우선 시장가/단순 주문
